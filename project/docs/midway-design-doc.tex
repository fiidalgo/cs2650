\section{Introduction}
This design document introduces my work on a Log-Structured Merge Tree (LSM-tree) based key-value store for the CS2650 NoSQL Systems project. 
The overarching problem is to design a storage engine that can efficiently handle high write throughput and still perform queries at scale. 
Such a system is important in real-world applications (e.g., time-series logging, NoSQL databases) where write performance and eventual consistency are often more critical than immediate in-place updates, and where large volumes of data must be ingested while remaining queryable.

\section{Challenges}
Designing an LSM-tree has the following key challenges:
\begin{enumerate}
    \item \textbf{Write Amplification vs. Read Amplification:} 
    In a naive solution, we might simply keep an on-disk B-Tree. But repeated, small in-place writes quickly degrade performance (random I/O). An LSM-tree reduces write amplification by batching writes in memory and merging them periodically. However, this increases read amplification unless we manage multiple on-disk files cleverly (e.g. with compaction and Bloom filters).

    \item \textbf{Maintaining Sorted Data for Fast Range Queries:} 
    We want to store data in sorted order for quick merges and range scans, but also want quick inserts. 
    Using a skip list for the in-memory buffer is helpful; still, merging multi-GB data on disk is non-trivial.

    \item \textbf{Expected Optimal Benefit:} 
    If perfectly tuned, an LSM-tree can ingest writes at near the speed of sequential I/O, while providing reasonably fast lookups and range queries with minimal overhead. 
    Achieving that ideal requires carefully orchestrated compactions and indexing (like fence pointers or Bloom filters).

    \item \textbf{Observations/Intuitions:} 
    Storing updates in an in-memory skip list, then flushing to on-disk Sorted String Tables (SSTables), will drastically speed up writes. 
    Searching from newest to oldest SSTables ensures the latest version is found first, but to keep read costs low, we will need future optimizations such as Bloom filters or better compaction policies.
\end{enumerate}

\section{Technical Descriptions}

\subsection{Current Design \& Implementation}
\textbf{In-memory buffer (MemTable).}
I use a skip list that stores \texttt{(key, value, is\_deleted)} records in sorted order. 
When the MemTable's size exceeds 4\,MB or 1\,million entries, it becomes immutable and is flushed to disk by a background thread. 
A fresh MemTable is allocated to handle new writes.

\textbf{On-disk storage (SSTables).}
Each flush produces an SSTable containing a header (format version, min/max key, etc.) followed by sorted records. 
Currently, Level 0 can have multiple SSTables; compaction to deeper levels is not yet implemented. 
For \texttt{get(k)}, the system looks in the active MemTable, any immutable MemTables, then scans the Level 0 SSTables (in reverse order) until it finds the key.

\subsubsection{Experiment 1: PUT Performance}
\paragraph{Goal and Setup.}
I generated varying numbers of \texttt{(key, value)} pairs (from 100K up to 1M records) and loaded them via the \texttt{load} command. 
I recorded total ingestion time and I/O stats.

\paragraph{Result Graph.}
Figure~\ref{fig:put-result} plots total PUT time for different record counts. It shows that time grows roughly linearly with the input size. Each flush triggers the creation of a new SSTable, which increases disk writes.

\paragraph{Explanation Graph.}
Figure~\ref{fig:put-io} shows how the write I/O also grows as more records are handled. Each new SSTable flush writes out an entire MemTable. Higher record counts lead to multiple flushes.

\section{Conclusion and Future Steps}
So far, I have built a functional LSM-tree that can handle PUT, GET, DELETE, RANGE, and basic flushes to on-disk SSTables. 
The experiments confirm that the current design scales linearly with data size and number of queries, but read amplification is high without compaction or Bloom filters. 

After the midway check-in, the plan is:
\begin{itemize}
    \item Implement multi-level compaction and a leveling or tiering policy to reduce the number of SSTables.
    \item Integrate Bloom filters and fence pointers to reduce read I/O on \texttt{get} operations.
    \item Introduce concurrency and multi-threaded flushing and compaction to further improve throughput.
\end{itemize}

These enhancements aim to bring the LSM-tree closer to the theoretical benefits of write-optimized data structures while keeping read costs practical.
